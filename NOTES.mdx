# 📅 Calendar Clone — Project Notes

> **Date:** April 12, 2026
> **Goal:** Build a Google Calendar clone to portfolio-level quality, targeting FAANG-calibre engineering standards.

---

## 🗂️ What Is This Project?

A **full-stack Google Calendar clone** built as a TypeScript monorepo. It replicates the core features of Google Calendar — multiple views, recurring events, calendar management, event search, dark mode, ICS import/export — with a clean production-ready architecture.

---

## 🏗️ Architecture Overview

```
calendar-clone/
├── apps/
│   ├── web/        ← Next.js 16 frontend (React 19, Tailwind CSS v4)
│   └── api/        ← NestJS 11 backend (Prisma, PostgreSQL)
├── package.json    ← Root monorepo config
├── pnpm-workspace.yaml
├── turbo.json
└── docker-compose.yml / docker-compose.prod.yml
```

### Why a monorepo?

- **Single repo, two apps** — frontend and backend live together, making it easy to share types and run both with one command (`pnpm dev`)
- **Turborepo** handles the build pipeline — it caches builds, runs tasks in the right order (build API before web if needed), and supports `--parallel` dev
- **pnpm workspaces** manages dependencies across packages efficiently with hard links (no duplicate `node_modules`)

## 🗄️ Database Schema

Four models, designed to match Google Calendar's data model exactly.

### `User`
```prisma
model User {
  id        String     @id @default(cuid())
  email     String     @unique
  googleSub String?    @unique   // Google's "sub" claim — stable user identifier
  name      String?
  calendars Calendar[]
  createdAt DateTime   @default(now())
  updatedAt DateTime   @updatedAt
}
```

**Why `cuid()` not `uuid()`?** CUIDs are URL-safe, shorter, and collision-resistant without a UUID generator.
**Why `googleSub`?** Google's `sub` claim is the only truly stable identifier — email can change.

---

### `Calendar`
```prisma
model Calendar {
  id      String  @id @default(cuid())
  name    String
  color   String?       // hex color, e.g. "#039BE5"
  ownerId String
  owner   User    @relation(...)
  events  Event[]

  @@index([ownerId])    // fast lookup of all calendars for a user
}
```

**Why a separate Calendar model?** Mirrors Google Calendar — one user can have multiple calendars (Personal, Work, Birthdays…), and events belong to a specific calendar.

---

### `Event`
```prisma
model Event {
  id          String   @id @default(cuid())
  calendarId  String
  title       String
  description String?
  location    String?

  allDay      Boolean  @default(false)

  // Timed events (stored as UTC)
  startAt     DateTime
  endAt       DateTime

  // All-day events (stored as DATE, not TIMESTAMP)
  startDate   DateTime? @db.Date
  endDate     DateTime? @db.Date   // exclusive end (Google Calendar convention)

  timeZone    String?

  color              String?
  recurrenceRule     String?   // RRULE string, e.g. "FREQ=WEEKLY;BYDAY=MO"
  recurrenceTimeZone String?
  guests             Json?
  notifications      Json?
  visibility         String?   // "public" | "private" | "default"
  busyStatus         String?   // "busy" | "free"

  @@index([calendarId, startAt])
  @@index([calendarId, startDate])
}
```

**Key design decisions:**

- `startAt`/`endAt` are always stored in **UTC** — the `timeZone` field tells you what timezone to *display* them in
- All-day events use `@db.Date` columns (PostgreSQL `DATE` type) — no time component, no timezone confusion
- `endDate` is **exclusive** — same convention as Google Calendar (an event "Jan 1–2" has `endDate = Jan 3`)
- `recurrenceRule` stores only the rule part (`FREQ=DAILY;COUNT=5`), not the `RRULE:` prefix
- `guests` and `notifications` are `Json?` — flexible without extra join tables

---

### `EventRecurrenceException`
```prisma
model EventRecurrenceException {
  id             String   @id @default(cuid())
  eventId        String
  originalStartAt DateTime  // identifies which occurrence this exception is for

  cancelled      Boolean  @default(false)  // true = this occurrence is deleted

  // All fields are nullable — null means "use the master event's value"
  title          String?
  description    String?
  startAt        DateTime?
  endAt          DateTime?
  // ... etc

  @@unique([eventId, originalStartAt])   // one exception per occurrence
}
```

**Why this pattern?** This is how Google Calendar handles "edit this event only" — instead of creating a new event, we store a diff against the master series. The `@@unique` constraint ensures no duplicate exceptions for the same occurrence.

## 🔐 Authentication

### The Full OAuth Flow

```
User → clicks "Sign in with Google"
  → GET /v1/auth/google/start          (API redirects to Google)
  → Google consent screen
  → GET /v1/auth/google/callback?code= (API handles this)
  → API verifies ID token with google-auth-library
  → API upserts User in DB (by googleSub)
  → API signs JWT, sets HttpOnly cookie
  → API redirects back to web app (http://localhost:3000)
  → Web app reads /v1/auth/me to get user info
```

### Why HttpOnly cookies instead of localStorage?

| | HttpOnly Cookie | localStorage |
|---|---|---|
| XSS-safe | ✅ JS can't read it | ❌ Any script can steal it |
| CSRF risk | Mitigated with `sameSite: lax` | No CSRF risk |
| Auto-sent | ✅ Browser sends automatically | ❌ Must set Authorization header |

### Why the API owns the OAuth flow?

The frontend **never sees the access token or ID token** — it only receives a signed JWT inside an HttpOnly cookie. This means even if someone runs malicious JS in the browser, they cannot steal the Google credentials.

### `JwtCookieGuard`

A custom NestJS guard that:
1. Reads `req.cookies[COOKIE_NAME]`
2. Verifies the JWT with `jsonwebtoken`
3. Attaches `req.user = { sub, email, name }` for controllers to use

Applied at controller class level so it's impossible to forget on a new route.

### Why `CalendarPageClient` is just an orchestrator

The principle of **"thin components, fat hooks"**. The page component only:
1. Calls domain hooks
2. Wires events together (e.g. unauthorized → redirect)
3. Renders the shell

No business logic lives in the component itself. This makes it easy to test hooks independently and swap out UI without touching logic.

## 🪝 Custom Hooks

### `useCalendarNavigation`
**Where:** `lib/hooks/useCalendarNavigation.ts`

Syncs `view` (year/month/week/day) and `date` to **URL search params**. This means:
- The calendar is **deep-linkable** — you can share a URL and someone lands on the same view and date
- The **back button works** — browser history tracks view changes
- **No React state** for navigation — the URL is the state

### `useEventsApi`
**Where:** `lib/events/useEventsApi.ts`

Owns the full events data lifecycle:
- Fetches events for the current date range
- Provides `addEvent`, `updateEvent`, `removeEvent`
- Handles 401 → sets `unauthorized = true` (parent redirects to login)
- All mutations `await refresh()` after API calls

**Why not React Query / SWR?** Kept dependencies minimal for a portfolio project. The hook is a clean implementation of the same pattern.

### `useCalendarsApi`
**Where:** `lib/calendars/useCalendarsApi.ts`

Single source of truth for calendars. Manages:
- `calendars[]` — the list from API
- `visibleCalendarIds` — which ones are checked
- `toggleCalendar`, `createCalendar`, `updateCalendar`, `deleteCalendar`

**Why hoisted to `CalendarPageClient`?** Previously `Sidebar.tsx` fetched calendars internally. This meant the data wasn't available anywhere else. Hoisting it means:
- Event forms can show a calendar selector
- The events list can be filtered by visibility
- Single API call instead of duplicate fetches

### `useKeyboardShortcuts`
**Where:** `lib/hooks/useKeyboardShortcuts.ts`

Global `keydown` listener. Skips when focus is in an input/textarea/select. Implements Google Calendar's exact shortcuts:

| Key | Action |
|---|---|
| `t` | Today |
| `d` | Day view |
| `w` | Week view |
| `m` | Month view |
| `y` | Year view |
| `c` | Create event |
| `←` `p` `k` | Previous period |
| `→` `n` `j` | Next period |

### `usePopoverState`
**Where:** `lib/hooks/usePopoverState.ts`

Manages both the event detail popover and the "more events" day popover state — open/closed, anchor rect for positioning, which event is active.

Classic Google Calendar month grid. Each week row shows events in the cells. "All-day" and multi-day events get colored bars. A "+N more" button opens `DayEventsPopover`.

Single-column version of WeekView. Same timeline math.

## 🔁 Recurring Events

This is the most complex part of the backend.

### How it works

1. A recurring event is stored as **one row** in the `Event` table with a `recurrenceRule` (e.g. `FREQ=WEEKLY;BYDAY=MO,WE,FR`)
2. When you query `GET /v1/events?from=&to=`, the backend **expands** the rule using the `rrule` library to generate all occurrences in the date range
3. Each virtual occurrence gets a virtual ID: `{masterId}@{originalStartAt}` — e.g. `clx123@2026-04-14T09:00:00.000Z`
4. Exceptions (edited/deleted occurrences) are stored in `EventRecurrenceException` and applied on top

### Edit scope options

When you edit a recurring event, you choose:
- **This event** — creates an exception row for just this occurrence
- **This and following events** — splits the series (sets an `UNTIL` on the original rule, creates a new series from this point)
- **All events** — updates the master event row

### Why RRULE?

RFC 5545 (iCalendar standard) is the universal format for recurring events. Google Calendar uses it. Using the same standard means ICS import/export works correctly.

## 🔍 Event Search

**Frontend:** `SearchBar` component in `CalendarHeader`
- 300ms debounce — doesn't hit the API on every keystroke
- Dropdown shows event title, color dot, and formatted date
- Clicking a result opens the `EventPopover` directly

**Backend:** `GET /v1/events/search?q=`
- Case-insensitive `contains` query (maps to `ILIKE` in PostgreSQL) on `title`, `description`, and `location`
- Returns up to 20 results ordered by most recent
- **Critically placed before `GET /v1/events/:id`** in the controller — otherwise Express would treat the literal string `"search"` as an `:id` parameter

## 🛡️ Security

### `helmet`
Added to `main.ts`. Automatically sets ~12 HTTP security headers:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Strict-Transport-Security` (HSTS)
- `Content-Security-Policy`
- …and more

In development, CSP is disabled so Swagger UI can load inline scripts.

### Rate Limiting (`@nestjs/throttler`)
120 requests per minute per IP, applied globally via `APP_GUARD`. Prevents:
- Brute-force attempts
- API abuse / scraping
- Accidental infinite request loops from bugs

### Production-safe cookies
```typescript
// Before (wrong):
secure: false // hardcoded!

// After (correct):
secure: process.env.NODE_ENV === 'production'
```

In production (HTTPS), the cookie is only sent over encrypted connections.

## 🐳 Docker & Deployment

### Multi-stage Dockerfiles

Both apps use **multi-stage builds** — a software engineering best practice for production containers:

1. **Stage 1 (deps/builder):** Install all dependencies, compile TypeScript/Next.js
2. **Stage 2 (runner):** Copy only the compiled output + production `node_modules` — the final image is much smaller (no TypeScript compiler, no dev dependencies)

### Next.js Standalone Output

```typescript
// next.config.ts
output: 'standalone'
```

This makes Next.js produce a self-contained `server.js` + only the required `node_modules`. The Docker image goes from ~1GB to ~200MB.

### `docker-compose.prod.yml`

Orchestrates three services:
- `db` — PostgreSQL 16 with health check
- `api` — NestJS backend (waits for DB to be healthy before starting)
- `web` — Next.js frontend (waits for API)

All configuration via environment variables — no secrets in the image.

## 🚀 Running the Project Locally

### Prerequisites
- Node.js LTS (v22+)
- pnpm (`npm i -g pnpm`)
- Docker (for PostgreSQL)

### Steps

```bash
# 1. Install all dependencies
pnpm install

# 2. Start PostgreSQL

pnpm dev
```

- **API:** http://localhost:3001

3. Set **Authorised redirect URI:** `http://localhost:3001/v1/auth/google/callback`
4. Copy `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` into `apps/api/.env`

## 🧠 Key Engineering Decisions (for interviews)

### 1. URL as state for navigation
Using `useSearchParams` for view/date means the calendar is deep-linkable and the browser back button works — no extra state management needed.

### 2. HttpOnly cookie auth (not localStorage)
A conscious security decision. Eliminates the entire class of XSS token-theft attacks. The tradeoff is you need `credentials: 'include'` on every fetch and a CORS policy.

### 3. `apiFetch` throws, never returns `{ok}`
Every API call either succeeds or throws `ApiError`. Callers use `try/catch`. This is more ergonomic than checking `.ok` at every call site and forces errors to be handled.

### 4. Hoisting state up
When `visibleCalendarIds` lived inside `Sidebar`, it was trapped — the events list couldn't see it. Hoisting it to `CalendarPageClient` made it available everywhere. This is the fundamental React "lifting state up" principle applied to a real problem.

### 5. Thin controllers, fat services (NestJS)
Controllers are 1-2 lines each. All business logic is in services. This makes services unit-testable without spinning up HTTP, and makes the API contract clear.

### 6. Multi-stage Docker builds
Final production images don't contain TypeScript, the compiler, or dev dependencies. The API image is ~3x smaller than a naive single-stage build.

### 7. Recurring events via exception model
Instead of creating N copies of an event, we store one master + diffs. This is exactly how Google Calendar, Outlook, and iCal work. It's efficient (one row vs potentially thousands) and correct (editing "all events" just updates one row).
